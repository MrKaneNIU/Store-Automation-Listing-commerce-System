# 高端女装商城管理工作台 UI PRD

## 0. 文档定位

本文档记录 2026-05-13 已确认的 V4 管理工作台方案，作为 `docs/prd/2026-05-12-high-fashion-mall-ui-reference-prd.md` 的管理端补充 PRD。

本 PRD 只定义管理工作台的高定女装商城风格、信息架构、功能映射和后续模块化执行顺序。当前不进入真实小程序迁移，不修改业务逻辑、数据结构、接口、路由配置或依赖。

V4 定稿结论：管理工作台不能做成传统后台表格，而要延续客户商城的黑白、高留白、圆角卡片、轻阴影和底部导航风格。首页以“管理工作台”为业务入口，快速进入当前项目已有的老板端、店员端和商家订单能力。

## 1. 背景与问题

当前高端女装商城原型已经完成客户侧视觉方向，但管理工作台仍缺少同风格页面。用户确认客户商城首页右上角的“管理”应作为管理工作台入口，并要求管理工作台也使用同一套 UI 参数、同一套高级女装商城气质，而不是跳到传统后台页面。

用户已经确认 V4 版本可定稿。V4 的核心变化是：

1. 客户首页右上角保留“管理”入口。
2. 管理端右上角增加“商城”入口，用于回到客户商城。
3. 管理端底部导航与客户商城底部导航保持同尺寸、同高度、同圆角气质。
4. 工作台首页保留“业务流进度”模块和三个高频功能入口。
5. 商品管理与订单确认迁入管理端底部导航，点击后独占页面。
6. 页面只展示项目现有能力，不发明新的后台功能。

## 2. 目标

1. 建立一个可按模块执行的管理工作台 UI PRD。
2. 让后续执行严格先做 HTTP 原型，再由用户确认是否迁移到真实小程序。
3. 让管理端视觉与客户商城一致：黑白、高留白、圆角卡片、轻阴影、克制文字层级。
4. 用管理端首页承载老板/店员最常用的三条业务流：截图识别、草稿确认、店员补图。
5. 用底部导航承载管理端主页面：工作台、商品管理、订单确认。
6. 保持现有 OCR、商品、SKU、订单、库存、上传、授权和权限边界不变。

## 3. 非目标

本 PRD 不做以下内容：

1. 不新增真实管理后台表格系统。
2. 不新增商品、SKU、订单、库存、收藏、购物袋、地址、支付、优惠券等数据模型。
3. 不改变现有老板端、店员端、客户侧功能语义。
4. 不改 `src/domain/`、`src/features/`、`src/services/`、`backend/`、`cloudfunctions/`。
5. 不改 `src/pages.json`、`package.json`、`pnpm-lock.yaml`。
6. 不接入真实后端、真实 CloudBase 调用、真实上传、真实 OCR、真实微信授权或真实订单创建。
7. 不把 V4 视觉伴随原型中的 `.superpowers/brainstorm/` 文件当作产品交付物。
8. 不在用户确认前把管理工作台视觉写入真实小程序页面。

## 6. V4 信息架构

### 6.1 客户商城到管理端

客户商城首页右上角保留一个小号“管理”按钮。

规则：

1. “管理”不进入客户侧底部导航。
2. “管理”入口不抢占客户商品浏览路径。
3. 点击“管理”后切换到管理工作台原型视图。
4. 原型阶段只做页面切换，不触发真实登录、权限、后端或路由写入。

### 6.2 管理端到客户商城

管理工作台右上角增加“小号商城”按钮。

规则：

1. 位置和尺寸参考客户首页右上角“管理”按钮。
2. 文案为“商城”。
3. 点击后回到客户商城首页原型视图。
4. 不使用返回箭头替代“商城”，避免用户不清楚当前是在管理端。

### 6.3 管理端底部导航

管理端底部导航与客户商城底部导航保持同尺寸、同高度、同圆角、同阴影气质。

底部导航固定为三个主项：

```text
工作台
商品管理
订单确认
```

规则：

1. `工作台` 显示 V4 管理工作台首页。
2. `商品管理` 独占一个管理端页面，不再作为工作台首页功能卡。
3. `订单确认` 独占一个管理端页面，不再作为工作台首页订单卡片。
4. 底部导航不承载 `截图识别`、`草稿确认`、`店员补图`，这三项只在工作台首页的业务流模块中出现。

## 7. V4 工作台首页规格

### 7.1 顶部信息

顶部使用同客户商城一致的高级电商气质。

文案：

```text
Oh My Fish
管理工作台
```

规则：

1. `Oh My Fish` 作为品牌/kicker。
2. `管理工作台` 必须单行显示。
3. `管理工作台` 不做传统后台大标题，不使用沉重 Dashboard 标题样式。
4. 顶部右侧固定显示“商城”入口。

### 7.2 业务流进度模块

业务流进度模块是首页主模块，不显示“入口保留”。

模块内显示三条业务节点：

| 节点 | 指标文案规则 | 交互目标 |
| --- | --- | --- |
| 截图识别 | `可上传 X 张` | 进入截图识别入口 |
| 草稿确认 | `待确认 N 条` | 进入草稿确认入口 |
| 店员补图 | `待补图 M 件` | 进入店员补图入口 |

截图识别指标规则：

```text
maxUploadCount = 18
selectedCount = 当前已选择或已上传截图数量
remainingCount = max(0, 18 - selectedCount)
显示文案 = 可上传 remainingCount 张
```

示例：

```text
已上传 0 张 -> 可上传 18 张
已上传 5 张 -> 可上传 13 张
已上传 18 张 -> 可上传 0 张
```

规则：

1. V4 不再显示“入口保留”。
2. 指标必须是业务状态摘要，不是说明性占位文字。
3. 原型阶段可用静态样例数值，但文案规则必须与真实能力对齐。
4. 后续真实迁移时，数值必须来自现有页面 ViewModel 或 feature 查询能力，不允许页面直接写入 repository 或绕过既有服务边界。

### 7.3 三个功能入口

工作台首页只保留三个高频功能入口：

```text
截图识别
草稿确认
店员补图
```

规则：

1. 三个入口归属于“业务流进度”模块下方。
2. 不再保留五个功能入口卡。
3. 不在首页展示 `商品管理` 入口卡。
4. 不在首页展示 `订单确认` 入口卡。
5. 入口卡使用白底、圆角、轻阴影、高留白风格。
6. 入口卡可以用短标题、简短状态和轻量箭头表达可点击，不使用传统后台按钮堆叠。

## 8. 商品管理页面规格

商品管理从工作台首页迁入管理端底部导航，点击后独占页面。

目标：

1. 展示现有商品管理能力。
2. 保持高端女装商城风格，不做传统后台表格首页。
3. 用卡片、筛选胶囊、状态摘要展示商品状态。

内容建议：

1. 顶部品牌/kicker 与标题。
2. 状态筛选：待补图、可上架、已上架。
3. 商品卡片列表：商品图、货号、商品名、状态、价格区间、上架操作入口。
4. 批量操作仅在现有能力支持时展示。

必须遵守：

1. 不新增商品字段。
2. 不新增 SKU 规则。
3. 不改变上架判断。
4. 不让页面直接操作数据层。

## 9. 订单确认页面规格

订单确认从工作台首页迁入管理端底部导航，点击后独占页面。

目标：

1. 展示现有商家订单确认能力。
2. 让订单处理也延续高定商城卡片式视觉。
3. 支持确认订单和取消订单的现有语义。

内容建议：

1. 顶部品牌/kicker 与标题。
2. 待确认订单摘要。
3. 订单卡片：客户信息、商品摘要、规格、数量、订单状态、金额。
4. 主要操作：确认订单。
5. 次要操作：取消订单。

必须遵守：

1. 不新增支付、物流、退款、优惠券能力。
2. 不改变订单状态流。
3. 不改变库存扣减与审计规则。
4. 不在客户侧个人中心或客户页面重复展示管理端订单卡。

## 10. 视觉规范

管理端必须继承客户商城 UI PRD 的核心参数。

### 10.1 风格关键词

```text
黑白
高留白
圆角卡片
轻阴影
克制字号
触控友好
电商感
非传统后台
```

### 10.2 色彩

```text
页面背景：#F8F8F8
主内容背景：#FFFFFF
卡片背景：#FFFFFF
主文字：#222222
弱文字：#9A9A9A
分割线：#E8E8E8
主按钮/黑卡：#050505
主按钮文字：#FFFFFF
```

禁止：

1. 紫蓝渐变。
2. 霓虹光效。
3. 重型后台深色仪表盘。
4. 大面积表格灰底。
5. 纯营销落地页 hero。

### 10.3 间距与圆角

沿用客户商城参考：

```text
页面左右安全边距：32rpx
区块间距：48rpx - 72rpx
卡片内边距：28rpx - 40rpx
卡片圆角：24rpx - 32rpx
胶囊按钮圆角：999rpx
底部导航高度：144rpx - 168rpx
底部导航顶部圆角：32rpx - 40rpx
```

### 10.4 交互状态

原型和真实迁移都必须考虑：

1. 点击态：轻微缩放或透明度变化。
2. 当前 tab 激活态：黑色文字或黑色图标。
3. 未激活 tab：弱灰文字。
4. 空状态：大留白、短文案、一个明确入口。
5. 加载态：使用与卡片比例一致的骨架，不用全屏转圈替代。
6. 错误态：用短文案说明下一步动作。

## 11. 业务边界

以下合同在所有模块中保持不变：

1. 客户浏览商品不触发登录。
2. 只有下单时触发微信登录和手机号授权。
3. 取消授权不创建订单。
4. 客户侧只展示已上架商品。
5. 商品、SKU、订单、库存、OCR、上传、权限、授权的数据结构不变。
6. 页面层不直接写 repository、mockDb 或 CloudBase 集合。
7. 原型阶段不调用真实服务。
8. 后续真实迁移必须继续通过现有页面 ViewModel、feature use-case 或 page-safe 查询能力接入。
