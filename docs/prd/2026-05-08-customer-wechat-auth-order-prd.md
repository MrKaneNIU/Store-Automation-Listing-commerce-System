# 客户侧微信快捷登录与手机号授权下单 PRD

## 1. 背景

昨日商城 MVP 已跑通 Mock OCR 到商家确认订单的完整闭环。人工验收时发现：客户侧商品详情页仍要求用户手动填写姓名和手机号，这不符合微信小程序真实使用习惯。

本轮目标是在不破坏既有主链功能的前提下，将客户下单身份来源从“手填姓名/手机号”改为“微信快捷登录 + 微信绑定手机号授权”。

## 2. 产品目标

客户进入小程序后，可以无登录浏览已上架商品；当客户点击下单时，系统判断登录和手机号授权状态：

```text
已登录且已有手机号 -> 直接进入下单流程
未登录 -> 弹出登录窗口 -> 登录成功后继续判断手机号
已登录但无手机号 -> 弹出手机号授权窗口 -> 授权成功后创建订单
授权取消或失败 -> 不创建订单，停留在商品详情页
```

MVP 当前不接真实后端，先用 Mock 微信登录和 Mock 微信手机号授权跑通流程。后续真实后端上线时，只替换 `services/auth` 里的实现，不重写页面主流程。

## 3. 本轮明确不做

1. 不接真实微信后端。
2. 不调用真实 `code2Session`。
3. 不调用真实手机号换取接口。
4. 不接微信支付。
5. 不做正式用户中心。
6. 不做登录态跨设备同步。
7. 不修改老板端 OCR、草稿确认、商品创建、补图、上架流程。
8. 不修改店员端补图流程。
9. 不改变现有库存扣减、防超卖和订单状态保护规则。

## 4. 关键约束

1. 商品浏览必须保持无登录门槛。
2. 只有点击下单时才触发登录/授权判断。
3. 商品详情页不再展示姓名和手机号输入框。
4. 订单创建必须继续经过现有 `mallWorkflow.createOrder` 或等价业务编排，避免绕开库存校验。
5. 客户手机号来源必须来自客户授权服务，不允许页面写死手机号。
6. Mock 微信授权必须封装在 `services/auth`，不允许写死在页面组件中。
7. 后续真实微信授权只能替换 Auth Service，不允许重写商品详情页主流程。
8. 本轮只能新增客户授权相关模型和服务，不能重构无关 domain。
9. 商家订单页仍需能看到客户手机号。
10. 如果客户取消登录或手机号授权，不能创建订单，也不能扣库存。

## 5. 用户流程

### 5.1 浏览商品

```text
客户打开小程序
-> 查看商品列表
-> 点击商品
-> 打开商品详情
```

要求：

- 未登录客户也可以浏览。
- 页面不得主动弹登录窗口。
- 页面不得要求客户先授权手机号才能看商品。

### 5.2 已登录客户下单

```text
选择规格
-> 点击下单
-> 发现已有登录态和手机号
-> 创建待商家确认订单
-> 扣减本地 SKU 库存
```

要求：

- 不重复弹登录窗口。
- 订单记录客户手机号。
- 商品库存不足时仍禁止下单。

### 5.3 未登录客户下单

```text
选择规格
-> 点击下单
-> 弹出登录窗口
-> 客户确认登录
-> 获取 Mock openid/customerId
-> 判断手机号授权状态
-> 如无手机号，继续弹手机号授权窗口
-> 授权成功后创建订单
```

要求：

- 登录成功后继续原下单动作，不要求客户重复点击。
- 登录失败或取消时，不创建订单。
- 登录失败或取消时，不扣库存。

### 5.4 已登录但未授权手机号客户下单

```text
选择规格
-> 点击下单
-> 发现已登录但无手机号
-> 弹手机号授权窗口
-> 授权成功后创建订单
```

要求：

- 不重复登录。
- 手机号授权失败或取消时，不创建订单。
- 手机号授权失败或取消时，不扣库存。

## 6. 数据设计

### 6.1 CustomerSession

建议新增客户授权会话模型：

```text
customerId: string
openid: string
phoneNumber?: string
nickname?: string
authSource: 'mock_wechat' | 'wechat'
loggedInAt: string
phoneAuthorizedAt?: string
```

说明：

- `customerId` 是本系统客户 ID。
- `openid` 是微信身份标识。MVP 阶段使用 Mock 值。
- `phoneNumber` 只有手机号授权成功后才存在。
- `authSource` 用于区分 Mock 和真实微信授权。

### 6.2 Order 字段兼容策略

为了不破坏老板订单页和既有测试，本轮优先采用兼容策略：

```text
customerName: '微信用户'
customerPhone: session.phoneNumber
customerId?: session.customerId
customerAuthSource?: session.authSource
```

约束：

- 订单页面仍展示手机号。
- 本轮不强制删除 `customerName/customerPhone`，避免扩大改动面。
- 后续接真实后端时，再统一清理订单客户字段。

## 7. 服务边界

### 7.1 Auth Service 接口

建议新增：

```text
src/services/auth/wechat-auth-service.ts
src/services/auth/mock-wechat-auth-service.ts
```

接口能力：

```text
getCurrentSession()
login()
authorizePhoneNumber()
logout()
```

约束：

- 页面层只能调用 Auth Service。
- 页面层不能直接生成 `openid`、`customerId`、`phoneNumber`。
- Auth Service 当前使用内存 Mock，后续可以替换真实微信后端实现。

### 7.2 Mock 微信授权行为

Mock 登录返回：

```text
customerId: mock-customer-001
openid: mock-openid-001
nickname: 微信用户
authSource: mock_wechat
```

Mock 手机号授权返回：

```text
phoneNumber: 13800000000
phoneAuthorizedAt: 当前时间
```

Mock 授权窗口可以先用页面内弹窗或状态提示模拟。

## 8. 页面改造范围

### 8.1 客户商品详情页

必须改：

- 删除姓名输入框。
- 删除手机号输入框。
- 下单按钮文案改为“微信手机号下单”或等价文案。
- 点击下单时走授权判断。
- 授权成功后继续创建订单。

不能改：

- 商品详情展示主结构。
- 商品规格选择规则。
- 库存不足判断。
- 已上架商品可见性规则。

### 8.2 客户商品列表页

本轮默认不改。

如确实需要展示登录状态，只能做轻量展示，不允许强制登录。

### 8.3 老板订单页

原则上不改。

如果订单字段新增 `customerId/customerAuthSource` 导致类型展示需要兼容，只允许做最小适配，不能重做订单页 UI。

## 9. 任务拆分

### 模块 1：客户授权模型与 Mock Auth Service

目标：建立客户登录/手机号授权服务边界。

任务：

1. 新增 `CustomerSession` 类型。
2. 新增 Auth Service 接口。
3. 新增 Mock Auth Service。
4. 支持获取当前登录态。
5. 支持 Mock 登录。
6. 支持 Mock 手机号授权。

验收标准：

- 未登录时 `getCurrentSession()` 返回空。
- Mock 登录后有 `customerId/openid/authSource`。
- Mock 手机号授权后有 `phoneNumber`。
- Auth Service 不依赖页面组件。

### 模块 2：订单创建参数兼容

目标：让订单可以记录微信授权客户信息，同时不破坏现有订单流程。

任务：

1. 扩展订单创建参数，支持 `customerId/customerAuthSource`。
2. 保留 `customerName/customerPhone` 兼容老板订单页。
3. 确保库存扣减、防超卖、取消释放库存不变。
4. 补充测试覆盖微信授权客户下单。

验收标准：

- 订单能记录 Mock 微信手机号。
- 订单仍为 `pending_merchant_confirm`。
- 下单仍扣减库存。
- 授权失败时不创建订单、不扣库存。

### 模块 3：客户商品详情页下单流程改造

目标：移除手填表单，改为登录/手机号授权后下单。

任务：

1. 删除姓名和手机号输入。
2. 新增下单前授权判断。
3. 未登录时弹出登录确认。
4. 已登录无手机号时弹手机号授权确认。
5. 授权成功后继续原下单动作。
6. 授权取消时阻断下单。

验收标准：

- 未登录客户能浏览商品详情。
- 商品详情页无姓名/手机号输入框。
- 未登录点击下单会触发登录提示。
- 登录和手机号授权完成后能创建订单。
- 已登录且有手机号时再次下单不重复弹窗。

### 模块 4：回归验证

目标：确认本轮改动不损伤昨天已验收主链。

任务：

1. 跑单元测试。
2. 跑类型检查。
3. 跑微信小程序构建。
4. 手工检查客户侧商品详情页下单流程。
5. 手工检查老板订单页仍能看到手机号。

验收标准：


全部通过。

## 11. 风险与防护

### 风险 1：误伤商品浏览

防护：

- 登录判断只能放在下单按钮触发后。
- 商品列表和商品详情进入时不能自动弹登录。

### 风险 2：绕开库存校验

防护：

- 授权完成后仍必须调用既有订单创建流程。
- 不允许页面直接写订单。

### 风险 3：授权取消仍创建订单

防护：

- Auth Service 需要明确返回失败或空状态。
- 页面下单逻辑必须在没有手机号时直接中断。

### 风险 4：真实微信接入返工

防护：

- 页面只依赖 Auth Service 接口。
- Mock 实现和真实实现保持同一接口。
- 不在页面中写死 Mock openid 或手机号。

### 风险 5：老板订单页失去手机号

防护：

- 本轮保留 `customerPhone` 字段。
- Mock 手机号授权成功后必须写入订单。

## 12. 后续真实后端接入预留

后续真实后端上线时，Auth Service 真实实现需要对接：

```text
wx.login
POST /api/wechat/login
open-type=getPhoneNumber
POST /api/wechat/phone
```

真实流程：

```text
小程序 wx.login 获取 code
-> 后端 code2Session 换 openid
-> 用户授权手机号获取 phone code
-> 后端调用微信接口换手机号
-> 后端创建或更新 customer
-> 前端拿到 customer session 后下单
```

本轮不实现上述真实接口，只保留服务边界。
