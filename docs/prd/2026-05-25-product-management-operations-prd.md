# 商品管理运营能力补齐 PRD

## 0. 文档定位

本 PRD 定义“商品管理”从当前上架队列升级为可运营商品工作台的第一阶段能力。

它补充而不是替代以下既有文档：

- `docs/prd/2026-05-07-mall-mvp-prd.md`
- `docs/prd/2026-05-08-enterprise-launch-master-prd.md`
- `docs/prd/2026-05-19-enterprise-launch-risk-grouped-prd.md`

本 PRD 只约束商品管理、商品详情展示、SKU/库存运营和上架前校验，不扩展支付、物流、购物车、优惠券、客服、退款等能力。

## 1. 背景

当前商品管理页已经具备商品状态筛选、单个上架、批量上架能力，但它更像“发布队列”，还不是服装商城所需的商品运营工作台。

最近人工测试暴露出两个直接问题：

1. 客户进入商品详情后，能看到售罄规格，但规格、库存状态、下单可用性之间的展示链路不完整。
2. 商品详情红框位置的商品简介是页面固定文案，商品管理中没有编辑商品简介的能力。

进一步分析后，商品管理还缺少服装商城必备的三类能力：

1. 可售规格管理。
2. 库存编辑与库存状态管理。
3. 上架前库存和规格校验。

## 2. 目标

1. 让老板可以在商品管理中维护商品简介，并在客户商品详情页展示。
2. 让老板可以查看和维护商品的 SKU 规格、售价、库存和可售状态。
3. 让商品上架前必须经过规格、库存、图片、价格等必要校验。
4. 让客户侧商品详情能准确展示商品简介、已选规格、库存状态和售罄状态。
5. 继续保护既有订单、库存预占、防超卖、微信授权和页面边界规则。

## 3. 非目标

本 PRD 第一阶段不做以下内容：

1. 不做支付、购物车、物流、优惠券、售后或退款。
2. 不做云 e 宝实时库存同步。
3. 不做多仓库、多门店库存。
4. 不做复杂颜色/尺码矩阵生成器，第一版仍以现有 `spec` 字符串为基础。
5. 不改变客户下单授权规则。
6. 不允许页面层直接写 repository、CloudBase 集合或订单数据。
7. 不改变现有订单库存预占和取消释放库存语义。

## 4. 核心产品链路

目标链路：

```text
老板上传截图
-> OCR/人工复核草稿
-> 系统生成 SPU/SKU
-> 店员补图
-> 老板进入商品管理
-> 编辑商品简介
-> 维护规格、售价、库存、可售状态
-> 上架前校验
-> 商品发布到客户侧
-> 客户查看商品简介并选择规格
-> 有库存规格可下单，售罄规格可查看但不可下单
-> 下单后预占库存
-> 商家确认或取消订单
```

## 5. 功能范围

### 5.1 商品简介编辑

商品简介属于 SPU 级字段，不属于 SKU。

商品管理中应支持：

1. 查看商品简介摘要。
2. 点击“编辑简介”进入编辑态。
3. 输入或修改商品简介。
4. 显示字数限制，建议第一版限制 120 字。
5. 保存后刷新商品管理列表。
6. 客户商品详情页红框位置展示该商品简介。
7. 商品简介为空时，客户侧展示兜底文案，不出现空白。

客户侧展示要求：

1. 商品简介显示在商品名称和价格下方、规格选择上方。
2. 已上架商品读取真实商品简介。
3. 简介缺失时显示默认文案，例如“暂无商品简介，商家正在完善中。”

### 5.2 可售规格管理

商品管理中应支持查看商品下所有 SKU：

1. 规格名，例如 `黑色/XL`。
2. 售价。
3. 当前库存。
4. 可售状态。
5. 客户侧状态预览：可售、有库存、售罄、停售。

第一版建议支持以下操作：

1. 编辑规格展示名。
2. 编辑 SKU 售价。
3. 编辑库存。
4. 设置 SKU 可售或停售。
5. 新增漏识别 SKU。
6. 删除未售卖且无订单关联的错误 SKU。

### 5.3 库存编辑

库存编辑必须保持可审计。

商品管理中应支持：

1. 单个 SKU 调整库存。
2. 批量补货。
3. 批量清零。
4. 保存调整原因，例如“盘点修正”“补货入库”“人工纠错”。
5. 写入库存流水或审计记录。

库存编辑约束：

1. 库存不能小于 0。
2. 已被订单预占的库存不能被页面直接绕过。
3. 人工调整不能破坏现有下单防超卖规则。
4. 售罄 SKU 可以在客户侧被选中查看状态，但不可下单。

### 5.4 上架前校验

点击单品上架或批量上架前，系统必须执行商品级校验。

必过条件：

1. 商品有主图。
2. 商品至少有 1 个 SKU。
3. 商品至少有 1 个可售且库存大于 0 的 SKU。
4. 所有可售 SKU 售价必须大于 0。
5. 所有可售 SKU 规格名不能为空。
6. 同一商品下不能存在重复规格。

阻断提示示例：

- “缺少主图，无法上架”
- “没有可售规格，无法上架”
- “全部规格暂无库存，请先补库存”
- “存在重复规格，请先合并或修改”
- “存在价格为 0 的规格，请先补全售价”

### 5.5 客户商品详情展示

客户商品详情应支持：

1. 展示商品简介。
2. 展示所有规格。
3. 点击有库存规格后，更新已选规格和库存状态。
4. 点击售罄规格后，更新已选规格和“暂无库存”状态。
5. 只有选中可售且有库存 SKU 时，才允许下单。
6. 售罄规格不能创建订单。

## 6. 数据与边界

### 6.1 商品字段

建议在 Product 主档中新增：

```text
description: string
```

字段语义：

1. 商品简介，SPU 级。
2. 默认为空字符串。
3. 客户侧展示时允许使用兜底文案。
4. 不参与 SKU 合并规则。
5. 不参与库存计算。

### 6.2 SKU 字段

第一版可复用现有字段：

```text
spec
salePrice
stock
```

如果进入可售状态管理，建议新增：

```text
isAvailable: boolean
```

若第一版暂不新增字段，可先用 `stock > 0` 表示可下单，用 `stock <= 0` 表示售罄。

### 6.3 页面边界

页面层必须继续只调用 page-facing ViewModel / Facade。

禁止：

1. 商品管理页直接写 `mockDb`。
2. 商品管理页直接写 repository。
3. 客户详情页直接计算复杂库存规则。
4. 页面层绕过订单创建流程。

## 7. 权限与安全

1. 商品简介、SKU、库存和上架操作只允许管理域用户执行。
2. 店员是否允许编辑库存，需按后续权限配置决定。
3. 客户侧只能读取已上架商品信息，不能调用商品管理接口。
4. 库存调整必须保留操作人、时间、原因和调整结果。
5. 错误提示不能泄露内部数据库或 CloudBase 细节。

## 8. 验收标准

### 8.1 商品简介

1. 当老板在商品管理中编辑商品简介时，系统应保存简介并在刷新后继续显示。
2. 当客户打开商品详情时，系统应在商品标题和规格选择之间展示该商品简介。
3. 当商品简介为空时，系统应展示兜底文案。
4. 当商品简介被修改时，客户侧再次进入详情应看到最新简介。

### 8.2 规格与库存

1. 当老板进入商品规格库存编辑时，系统应展示该商品所有 SKU。
2. 当老板修改库存并保存时，系统应更新库存并记录库存流水。
3. 当 SKU 库存为 0 时，客户侧应显示售罄或暂无库存。
4. 当客户选中售罄 SKU 时，系统应显示已选规格和暂无库存，但不得允许下单。
5. 当客户选中有库存 SKU 时，系统应显示库存状态并允许进入下单授权流程。

### 8.3 上架校验

1. 当商品没有主图时，系统应阻止上架。
2. 当商品没有 SKU 时，系统应阻止上架。
3. 当商品没有任何可售且有库存 SKU 时，系统应阻止上架。
4. 当商品存在重复规格时，系统应阻止上架并提示原因。
5. 当商品满足所有校验时，系统应允许上架。

## 9. 建议实施顺序

建议拆成三个小模块推进，不混在一轮里完成：

### 模块 A：商品简介可编辑

目标：补齐商品主档简介字段和客户详情展示。

范围：

1. Product 增加 `description`。
2. 商品创建默认简介为空。
3. 商品管理支持编辑简介。
4. 客户详情展示简介。
5. 补 facade、页面和 CloudBase/API 测试。

### 模块 B：规格库存编辑工作台

目标：让商品管理能维护 SKU 运营信息。

范围：

1. 商品卡片增加“规格库存”入口。
2. 展示 SKU 表。
3. 支持编辑规格名、售价、库存。
4. 支持补货和清零。
5. 库存调整写审计或库存流水。

### 模块 C：上架前校验

目标：让上架动作具备服装商城必要保护。

范围：

1. 增加上架校验规则。
2. 单品上架和批量上架都复用同一校验。
3. 阻断原因在商品管理页展示。
4. 客户侧只看到真正可售的上架商品。

## 10. 验证要求

### 10.1 工具与 Skill 准则

如果某个实现任务会修改 UI 结构、视觉样式、交互方式、页面布局、弹窗、表单、按钮状态或客户/老板可感知的界面体验，在调用该任务必须的工具和 skill 的前提下，还必须额外调用 `ui-ux-pro-max` skill 做 UI/UX 约束检查。

该准则适用于：

1. 商品简介编辑入口和编辑弹层。
2. SKU/库存编辑工作台。
3. 上架前校验提示。
4. 客户商品详情简介、规格、库存状态展示。

纯后端、纯数据迁移、纯测试或纯文档任务不需要额外调用 `ui-ux-pro-max`。

### 10.2 自动验证

每个模块实现后至少运行：

```powershell
pnpm.cmd run verify
```

涉及小程序页面或构建行为时，还必须运行：

```powershell
pnpm.cmd run verify:full
```

### 10.3 人工验收建议

1. 老板编辑商品简介，客户详情红框位置正确显示。
2. 老板把某 SKU 库存改为 0，客户侧可选中查看但不可下单。
3. 老板把全部 SKU 库存清零，商品无法通过上架前校验。
4. 老板补库存后重新上架，客户侧可选择规格并进入下单授权。

## 11. 风险

1. 商品简介字段需要贯穿本地 mock、CloudBase、后端和客户详情，容易漏一层。
2. 库存编辑如果绕过库存流水，会破坏 Phase 5 的审计目标。
3. 上架校验如果只做前端提示，不做 facade/API 层校验，会被批量上架绕过。
4. SKU 删除如果不检查订单关联，可能破坏历史订单展示。

## 12. 暂定决策

1. 商品简介优先作为第一个小模块实现。
2. SKU/库存编辑作为第二个模块单独实现。
3. 上架前校验作为第三个模块实现，并同时覆盖单品上架和批量上架。
4. 售罄 SKU 客户侧可以选中查看状态，但不能下单。
5. 商品管理增强不得改变订单、授权、库存预占和取消释放库存主链。
