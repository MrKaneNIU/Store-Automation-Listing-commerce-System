# 微信小程序服装商城 MVP PRD

## 1. 产品背景

目标用户是服装店老板。老板已经在云 e 宝 App 中完成商品入库和商品信息维护，不希望在微信小程序商城中重复录入商品。

本 MVP 的核心目标是：老板上传已经填写好商品信息的云 e 宝页面截图，系统通过 Mock OCR 先跑通识别和商品创建闭环，后续再替换真实 OCR/AI 服务。

产品长期边界：系统不接入云 e 宝 API，不读取云 e 宝账号数据，不保存云 e 宝账号密码。云 e 宝只作为截图来源，数据进入本系统的唯一方式是老板主动上传截图。

## 2. MVP 目标

用最小可用功能跑通以下闭环：

```text
批量上传云 e 宝已填写页面截图
-> Mock OCR 识别商品草稿
-> 老板查看、修改、删除、确认草稿
-> 系统按货号和规格自动创建商品与 SKU
-> 店员补商品图片
-> 老板上架商品
-> 客户浏览已上架商品并提交订单
-> 商家确认订单
```

## 3. 目标用户

- 老板：上传截图、确认识别结果、管理商品、上架商品、确认订单。
- 店员：查找待补图商品并上传商品图片。
- 客户：浏览已上架商品、选择规格、提交订单。

MVP 阶段先通过角色入口切换老板端、店员端和客户侧，不做正式登录权限。

## 4. 核心方案

采用方案 1：**识别云 e 宝已填写商品页面截图**。

上传截图必须来自云 e 宝中已经填写了商品信息的页面，例如商品编辑页、商品详情页、规格页或规格弹窗截图。

如果截图仍然是空白新增商品页，只显示“请输入”“请选择”等占位内容，系统只能识别字段标签，不能生成有效商品草稿。此类结果必须进入“待补全”状态，由老板手动补齐后才能确认。

## 5. OCR 识别字段

MVP 阶段 OCR 只识别以下字段：

| 字段 | 英文字段名 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| 商品货号 | `product_code` | 是 | 用于聚合 SPU |
| 商品名称 | `product_name` | 是 | 用于商城展示 |
| 销售价 | `sale_price` | 是 | 用于 SKU 售价和客户侧展示 |
| 规格 | `spec` | 是 | 用于生成 SKU |

MVP 阶段不强制识别颜色、尺码、库存。规格先作为一个整体文本处理，例如“黑色/M”“白色/L”“均码”。

## 6. 截图输入约束

1. 只支持云 e 宝已填写商品信息的页面截图。
2. 同一批次可以上传多张截图。
3. 系统不要求每张截图都包含全部字段。
4. 同一批次内，系统会尝试把识别结果合并为商品草稿。
5. 如果规格信息在单独页面、弹窗或列表中，老板需要额外上传规格截图。
6. 空白新增商品页不能直接生成可确认商品。
7. 截图中字段值模糊、缺失或只出现占位文案时，草稿必须标记为待补全。

## 7. 明确不做与长期约束

以下内容不仅 MVP 阶段不做，后续迭代也不作为默认方向：

- 不接入云 e 宝 API。
- 不爬取云 e 宝数据。
- 不保存云 e 宝账号密码。
- 不做云 e 宝实时库存同步。

MVP 阶段还不做以下内容：

- 不做微信支付。
- 不做物流、退款、优惠券、会员系统。
- 不做正式多角色权限系统。
- 不做复杂图片云存储，先保留可替换上传服务接口。
- 不做真实 OCR/AI 服务，先使用 Mock OCR 跑通闭环。

## 8. 数据规则

### 8.1 商品聚合规则

```text
同一 product_code -> 一个商品 SPU
同一 product_code + 同一 spec -> 一个 SKU
同一 SKU 重复出现 -> 合并库存
```

### 8.2 库存规则

MVP 阶段 OCR 不强制识别库存。

默认策略：

- 如果草稿中没有库存，默认库存为 1。
- 如果后续人工编辑填写库存，则以人工填写为准。
- 同一 SKU 重复出现时，库存累加。

### 8.3 价格规则

- `sale_price` 是必填字段。
- 同一商品货号下出现多个销售价时，系统保留草稿记录并提示价格冲突。
- 老板确认前必须处理明显价格冲突。

## 9. 状态流

### 9.1 OCR 批次状态

```text
uploaded -> recognized -> confirmed
```

### 9.2 商品草稿状态

```text
pending -> confirmed
pending -> deleted
pending -> needs_completion
```

### 9.3 商品状态

```text
pending_images -> ready_to_publish -> published
```

### 9.4 订单状态

```text
pending_merchant_confirm -> confirmed
pending_merchant_confirm -> canceled
```

## 10. 模块拆分

### 模块 1：老板端截图导入

目标：老板能批量上传云 e 宝已填写商品页面截图，并创建识别批次。

最小功能：

- 批量选择截图。
- 展示截图缩略图。
- 删除误选截图。
- 点击“开始识别”。
- 创建 OCR 批次。

验收标准：

- 无截图时不能开始识别。
- 多张截图进入同一批次。
- 每个批次有唯一编号。

### 模块 2：Mock OCR 草稿生成

目标：使用 Mock OCR 模拟识别结果，跑通后续流程。

最小功能：

- 输入截图批次。
- 输出商品草稿。
- Mock 数据包含正常数据、重复 SKU、缺字段、低置信度、价格冲突。
- 草稿字段限定为商品货号、商品名称、销售价、规格、库存默认值。

验收标准：

- 能生成草稿列表。
- 缺少必填字段的草稿进入待补全。
- Mock OCR 被封装在 OCR Provider 中，后续可替换真实 OCR。

### 模块 3：老板端草稿确认

目标：老板能查看、修改、删除、确认识别结果。

最小功能：

- 草稿按商品货号分组。
- 支持编辑商品货号、商品名称、销售价、规格、库存。
- 支持删除错误草稿。
- 支持低置信度提示。
- 支持缺字段提示。
- 支持价格冲突提示。
- 支持批量确认。

验收标准：

- 商品货号、商品名称、销售价、规格缺一不可。
- 待补全草稿不能直接确认。
- 已删除草稿不会进入商品创建。
- 确认后进入 SPU/SKU 创建流程。

### 模块 4：自动创建商品和 SKU

目标：按确认草稿生成商品和 SKU。

最小功能：

- 同一商品货号生成一个商品 SPU。
- 同一商品货号和规格生成一个 SKU。
- 重复 SKU 合并库存。
- 商品创建后进入待补图状态。

验收标准：

- 同一货号不会生成多个商品。
- 不同规格会生成不同 SKU。
- 重复 SKU 库存正确累加。
- 核心规则有自动化测试。

### 模块 5：店员补图

目标：店员能查找待补图商品并上传商品图片。

最小功能：

- 待补图商品列表。
- 按商品货号搜索。
- 按 OCR 批次筛选。
- 上传主图。
- 上传详情图。
- 有主图、销售价和 SKU 后，商品变为可上架。

验收标准：

- 无主图商品不能上架。
- 补主图后商品状态变为可上架。
- 店员不需要进入 OCR 草稿页面。

### 模块 6：老板上架

目标：老板能管理商品状态，并把可上架商品发布到客户侧。

最小功能：

- 商品列表。
- 状态筛选：待补图、可上架、已上架。
- 单个上架。
- 批量上架。

验收标准：

- 只有可上架商品能上架。
- 已上架商品客户侧可见。
- 待补图商品客户侧不可见。

### 模块 7：客户侧下单

目标：客户能浏览已上架商品、选择规格并提交订单。

最小功能：

- 商品列表。
- 商品详情。
- 规格选择。
- 库存展示。
- 填写姓名、手机号。
- 提交订单。

验收标准：

- 客户侧只展示已上架商品。
- 无库存 SKU 不可下单。
- 订单提交后状态为待商家确认。

### 模块 8：商家确认订单

目标：商家能查看并处理客户订单。

最小功能：

- 待确认订单列表。
- 订单详情。
- 确认订单。
- 取消订单。

验收标准：

- 新订单默认待商家确认。
- 确认后状态变为已确认。
- 取消后状态变为已取消。

## 11. 页面范围

### 老板端

- 老板工作台
- 截图导入
- 草稿确认
- 商品管理
- 订单确认

### 店员端

- 待补图商品列表
- 商品补图

### 客户侧

- 商品列表
- 商品详情
- 下单页
- 下单结果

## 12. 架构约束

1. 页面层只负责展示和交互。
2. OCR 逻辑必须通过 `services/ocr` 接入。
3. 上传逻辑必须通过 `services/storage` 接入。
4. 数据读写必须通过 repository 接入。
5. SPU/SKU 聚合、草稿校验、上架判断、下单判断必须放在 domain 或 rules 中。
6. Mock OCR 不允许写死在页面里。
7. 后续真实 OCR/AI 只能替换 OCR Provider，不重写页面流程。
8. 每个模块完成后必须能单独验收。

## 13. 验收总标准

MVP 完成时必须满足：

1. 老板能上传一批截图并生成草稿。
2. 老板能修正草稿并确认。
3. 系统能自动生成商品和 SKU。
4. 店员能为待补图商品补主图。
5. 老板能上架商品。
6. 客户只能看到已上架商品。
7. 客户能选择规格并提交订单。
8. 商家能确认订单。
9. 核心 SPU/SKU 规则有测试。
10. 构建目标 `mp-weixin` 通过。

## 14. 后续迭代

MVP 闭环通过后，再分阶段迭代：

1. 接入真实 OCR/AI 服务。
2. 增强规格解析，将 `spec` 拆成颜色、尺码等结构化字段。
3. 接入图片对象存储。
4. 增加登录和角色权限。
5. 增加订单库存扣减策略。
6. 接入微信支付。
7. 优化老板端批量编辑体验。
8. 优化客户侧商城 UI。
