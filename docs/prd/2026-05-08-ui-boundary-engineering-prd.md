# UI 边界工程化 PRD

## 1. 背景

当前微信小程序已经跑通商城主闭环，并完成了客户侧 Mock 微信登录与手机号授权下单。随后项目又加入了更严格的 harness engineering 约束：页面不能直接访问底层 repository，业务规则必须留在 domain 和 features，服务适配必须留在 services。

下一阶段会继续接入真实持久化、真实图片存储、真实微信授权、真实 OCR/AI 和角色权限。等这些生产底座完成后，项目还会进入大规模 UI 优化或 UI 重构。如果 UI 层边界不提前收紧，后续 UI 改版容易误伤库存、订单、草稿确认、上架、授权等已经验收过的业务链路。

本轮目标不是做视觉升级，而是先把 UI 层整理成稳定外壳：页面只负责展示、交互、跳转和轻量页面状态；业务编排、状态判断、数据查询、提交动作和提示数据应沉到 features 的 page-facing Facade 或 ViewModel 中。

## 2. 问题陈述

从产品继续迭代的角度看，当前主要问题是：

1. 页面已经不再直接访问 `mockDb` 或 `mallRepository`，但部分页面仍直接组合 `mallAccess`、`mallWorkflow`、业务 helper 和页面状态。
2. 一旦开始 UI 重构，开发者可能在页面中重新组织业务判断，导致已验收链路被无意改变。
3. 页面级错误提示、空状态、按钮禁用、筛选状态和提交反馈分散在 `.vue` 文件中，缺少可单独测试的 page-facing 模块。
4. 大规模 UI 改版前，还缺少明确的“页面可改区域”和“业务不可改区域”清单。
5. 现有测试重点覆盖 domain、workflow、repository、auth 和 OCR，但页面 ViewModel 行为仍是测试缺口。

## 3. 产品目标

本轮完成后，后续 UI 优化应满足：

```text
旧页面 UI
-> 调用稳定的 page-facing Facade / ViewModel
-> 新页面 UI
```

业务内核保持不变，UI 可以更容易替换。

具体目标：

1. 明确 UI 层只负责展示、交互、弹窗、toast、跳转和页面本地输入状态。
2. 为高风险页面建立 page-facing ViewModel 或 Facade，使页面不再直接拼装复杂业务数据。
3. 将页面所需的展示模型、按钮可用性、空状态、提示信息、提交动作入口收口到 features 层。
4. 保持现有业务闭环、字段、状态和服务接口不变。
5. 给后续大规模 UI 重构提供可回归验证的保护网。

## 4. 本轮明确不做

1. 不做视觉重设计。
2. 不更换 TDesign/Vant 或新增 UI 组件库。
3. 不修改商品、SKU、草稿、订单、客户会话的数据模型。
4. 不接真实数据库、真实图片存储、真实微信后端或真实 OCR。
5. 不改变老板端、店员端、客户侧任何已验收业务流程。
6. 不重写 `mallWorkflow` 的业务规则。
7. 不降低测试断言，不修改已验收 fixture 来适配重构。
8. 不新增跨页面隐藏全局状态。

## 5. 边界原则

### 5.1 页面层职责

`src/pages` 只允许负责：

- 页面布局和组件结构。
- 页面本地输入状态。
- 用户点击、选择、弹窗确认、toast、loading。
- 路由参数读取和跳转。
- 调用 features 暴露的 page-facing 方法。
- 将 ViewModel 返回的展示数据渲染出来。

页面层不允许负责：

- 生成 mock openid、customerId、手机号、OCR 行或上传 URL。
- 判断草稿是否可确认。
- 判断商品是否可上架。
- 判断库存是否足够或扣减库存。
- 判断订单状态是否可确认或取消。
- 直接访问 repository、mockDb、数据库或底层存储。
- 直接拼装跨模块业务对象。

### 5.2 Features 层职责

`src/features` 应承接页面需要的业务入口：

- 页面查询 Facade。
- 页面 ViewModel。
- 页面动作 Command。
- 筛选、分组、展示摘要、按钮可用性、错误消息映射。
- 对 workflow、domain 和 services 的编排。

Features 层可以调用 domain 和 services，但必须保持接口面向用例，而不是面向 UI 组件细节。

### 5.3 Domain 层职责

`src/domain` 继续负责纯业务规则：

- 草稿必填字段和完成状态。
- SPU/SKU 聚合。
- 商品上架资格。
- 订单创建和订单状态流转。
- 库存资格、扣减和恢复相关不变量。

本轮除非发现现有规则缺少测试，不主动修改 domain。

### 5.4 Services 层职责

`src/services` 继续负责可替换外部能力：

- OCR Provider。
- Upload Service。
- Auth Service。
- Repository 实现。

页面不得直接调用 mock service 实现。若当前页面为了演示仍需要 mock 实例，应在本轮改造中收口到 feature 层或 page-facing 组合入口。

## 6. 建议执行方案

本轮建议拆为 4 个模块，每个模块完成后停下验证。

### 模块 1：页面风险盘点与 UI 边界清单

目标：列出当前每个页面的调用关系和迁移优先级。

任务：

1. 扫描 `src/pages` 中所有 `.vue` 文件的 imports。
2. 标记页面直接调用的 feature、workflow、access、service 和 domain helper。
3. 按风险排序：
   - 高风险：客户商品详情页、老板草稿确认页、老板商品管理页、老板订单页、店员补图页。
   - 中风险：客户商品列表页、老板截图导入页。
   - 低风险：角色入口、静态导航页面。
4. 输出每个页面的“可保留页面逻辑”和“应下沉 features 逻辑”。

验收标准：

- 每个页面都有边界结论。
- 没有提出业务行为变化。
- 高风险页面都有下一步 ViewModel 或 Facade 目标。

### 模块 2：客户商品详情页 ViewModel

目标：保护客户下单链路，为后续商品详情 UI 改版做准备。

建议新增或调整 page-facing 模块，负责：

- 根据商品 ID 返回详情展示模型。
- 返回可选 SKU 列表和库存展示信息。
- 判断下单按钮是否可点击。
- 封装微信授权下单动作入口。
- 将授权取消、库存不足、商品不存在、未发布等结果映射成页面提示。

页面只保留：

- 规格选择状态。
- 点击事件。
- 弹窗或 toast 调用。
- 展示 ViewModel 返回的数据。

验收标准：

- 商品详情页不直接导入 mock auth service。
- 下单仍通过现有授权下单链路。
- 未登录可浏览商品详情。
- 授权取消不创建订单、不扣库存。
- 已授权客户下单仍写入手机号和授权字段。

### 模块 3：老板草稿确认页 ViewModel

目标：保护 OCR 草稿确认链路，为后续老板端 UI 重构做准备。

建议 ViewModel 负责：

- 获取最新批次。
- 获取并分组草稿。
- 计算待补全数量、低置信度提示、价格冲突提示。
- 提供草稿编辑、删除、替换和确认入口。
- 将确认失败原因转换为页面可展示消息。

页面只保留：

- 输入框绑定。
- 用户编辑事件。
- 确认按钮点击。
- 展示分组和提示。

验收标准：

- 页面不直接拼装确认所需业务数据。
- 待补全草稿仍不能确认。
- 已删除草稿不进入商品创建。
- 价格冲突提示仍保留。
- 重复确认仍不会重复创建商品和 SKU。

### 模块 4：老板商品、订单和店员补图页面 Facade 收口

目标：将其他高频页面的筛选、按钮状态和动作入口收紧。

建议覆盖：

- 老板商品管理页：
  - 商品状态筛选。
  - 可上架列表。
  - 单个上架和批量上架入口。
  - SKU 数量展示。
- 老板订单页：
  - 订单列表展示模型。
  - 可确认、可取消状态。
  - 确认和取消动作入口。
  - 客户手机号展示兼容。
- 店员补图页：
  - 待补图商品列表。
  - 批次筛选。
  - 补图动作入口。
  - 补图后状态展示。

验收标准：

- 页面不自己判断复杂业务状态。
- 商品上架资格仍由 domain/workflow 保护。
- 订单确认和取消仍保持状态保护。
- 补图后商品仍按既有规则进入可上架状态。

## 7. 用户故事

1. 作为老板，我希望后续页面变漂亮时商品草稿确认规则不变，所以我可以放心继续使用 OCR 导入流程。
2. 作为老板，我希望草稿缺字段、低置信度和价格冲突提示不因 UI 改版消失，所以我能继续发现识别风险。
3. 作为老板，我希望商品上架按钮只在规则允许时可用，所以不会误上架不完整商品。
4. 作为老板，我希望订单确认和取消仍受状态保护，所以不会重复处理订单。
5. 作为店员，我希望补图页面改版后仍只看到待补图商品，所以我不用进入老板的 OCR 和订单流程。
6. 作为客户，我希望商品详情页改版后仍能无登录浏览，所以浏览商品没有门槛。
7. 作为客户，我希望只有点击下单时才触发微信授权，所以不会一进页面就被打断。
8. 作为客户，我希望授权失败或取消时不会创建订单，所以不会产生误订单。
9. 作为开发者，我希望页面只调用稳定的 ViewModel 或 Facade，所以 UI 改版不用理解完整业务链。
10. 作为开发者，我希望 UI 重构前有页面边界测试，所以我能知道自己是否误伤业务行为。
11. 作为后续接真实后端的开发者，我希望页面不依赖 mock service，所以替换真实 auth、OCR、upload、repository 时不用改 UI 主流程。

## 8. 页面改造优先级

### 第一优先级

1. 客户商品详情页：牵涉授权、下单、库存预占，是客户侧最高风险页面。
2. 老板草稿确认页：牵涉 OCR 草稿、缺字段、价格冲突、SPU/SKU 创建，是老板端最高风险页面。

### 第二优先级

1. 老板商品管理页：牵涉上架资格和批量上架。
2. 老板订单页：牵涉订单状态保护和客户手机号展示。
3. 店员补图页：牵涉商品图片补全和商品状态变化。

### 第三优先级

1. 客户商品列表页：主要是展示和跳转。
2. 老板截图导入页：主要是截图选择和创建 OCR 批次。
3. 首页角色入口：主要是导航。

## 9. 测试策略

### 9.1 新增测试重点

本轮应优先增加 page-facing ViewModel / Facade 测试，而不是直接测试 `.vue` 内部实现。

测试应覆盖：

1. 客户商品详情 ViewModel：
   - 商品不存在。
   - 商品未上架。
   - SKU 缺货。
   - 未登录下单触发授权链。
   - 授权取消不创建订单。
   - 授权成功创建订单并扣减库存。
2. 老板草稿确认 ViewModel：
   - 最新批次为空。
   - 草稿按货号分组。
   - 缺字段草稿被标记。
   - 价格冲突提示存在。
   - 确认成功创建商品和 SKU。
3. 商品、订单、补图 Facade：
   - 上架按钮状态。
   - 批量上架只处理可上架商品。
   - 待确认订单可处理，非待确认订单不可处理。
   - 补图后商品状态进入可上架。

### 9.2 回归保护

实现本 PRD 后，必须继续保护完整主链：

```text
Mock OCR batch
-> draft validation
-> owner confirmation
-> product/SKU creation
-> image supplementation
-> product publish
-> customer authorized order
-> stock reservation
-> merchant confirm/cancel
```

必跑命令：

```powershell
pnpm.cmd run verify
```

如果改动触及页面、构建或小程序产物，还必须运行：

```powershell
pnpm.cmd run verify:full
```

## 10. 交付标准

本轮完成时必须满足：

1. 高风险页面有明确的 ViewModel 或 Facade 边界。
2. 页面不直接导入 mock service 实现。
3. 页面不直接访问 repository 或 mockDb。
4. 页面不新增业务规则判断。
5. 业务行为与当前验收基线一致。
6. 新增 ViewModel / Facade 有测试覆盖。
7. `pnpm.cmd run verify` 通过。
8. 若触及页面或构建产物，`pnpm.cmd run verify:full` 通过。
9. 文档同步说明后续 UI 重构应只替换页面结构，不改变 ViewModel/Facade 合约。

## 11. 风险与防护

### 风险 1：UI 边界工程化变成业务重构

防护：

- 每个模块只迁移页面组合逻辑。
- 不修改 domain 规则。
- 不修改订单、库存、草稿、商品字段。

### 风险 2：ViewModel 过度贴合当前 UI

防护：

- ViewModel 返回稳定展示模型和动作入口，不返回组件私有结构。
- 命名以业务场景为中心，不以当前 DOM 布局为中心。

### 风险 3：页面提示行为丢失

防护：

- 改造前列出现有空状态、错误提示、按钮禁用和成功提示。
- 改造后用测试覆盖关键提示条件。

### 风险 4：真实后端接入时再次返工

防护：

- 页面只依赖 feature/page-facing 合约。
- auth、OCR、upload、repository 的 mock 或真实实现继续留在 services。
- 禁止页面导入 mock adapter。

### 风险 5：验证成本过高

防护：

- 先从两个最高风险页面开始。
- 每个模块完成后运行窄测试，再运行 `verify`。
- 不在同一轮做视觉重设计。

## 12. 后续与 UI 重构的关系

本 PRD 是大规模 UI 重构前的前置工程。它完成后，后续 UI 优化可以按以下方式推进：

```text
稳定业务内核
-> 稳定 page-facing ViewModel / Facade
-> UI 组件和布局重构
-> verify:full + 人工主链验收
```

后续 UI 重构时，允许修改：

- 页面布局。
- 组件结构。
- 样式。
- 文案。
- 视觉层级。
- 空状态和加载状态展示方式。

后续 UI 重构时，默认不允许修改：

- domain 规则。
- workflow 行为。
- repository 合约。
- auth、OCR、upload service 接口。
- 商品、SKU、草稿、订单和客户会话字段含义。

## 13. 建议下一步

下一轮如果开始执行，建议只做第一批：

1. 模块 1：页面风险盘点与 UI 边界清单。
2. 模块 2：客户商品详情页 ViewModel。

完成后停下，运行验证并更新交付记录。确认客户下单链路不受影响后，再进入老板草稿确认页 ViewModel。
