# 管理工作台账号登录、更多模块与权限设置 PRD

## 0. 文档定位

本 PRD 定义管理工作台的账号密码登录、底部导航“更多”、权限管理、首页设置、账号管理四类能力。

它补充而不是替代以下既有文档：

- `docs/prd/2026-05-08-enterprise-launch-master-prd.md`
- `docs/prd/2026-05-13-admin-workbench-high-fashion-ui-prd.md`
- `docs/prd/2026-05-19-enterprise-launch-risk-grouped-prd.md`

本 PRD 只约束管理域，不扩展客户浏览、OCR、商品、订单、库存等既有业务链。

## 1. 背景

当前管理工作台缺少明确的账号密码登录门禁，也缺少面向管理域的权限管理和配置入口。

新的产品要求是：

1. 进入管理工作台前必须先弹出登录 UI。
2. 管理工作台不使用微信登录，改用账号密码登录。
3. 创作者初始账号为 `admin`，初始密码为 `123456`。
4. 创作者可以授权店铺老板，老板可以继续授权员工。
5. 底部导航新增“更多”。
6. “更多”页必须包含“权限管理”“首页设置”“账号管理”。
7. 登录 UI 采用参考图的结构与气质。

## 2. 目标

1. 建立一个独立的管理域账号体系。
2. 让工作台入口在无会话时先进入登录页。
3. 让登录态仅在本次小程序运行周期内保持，重新进入小程序后失效。
4. 让“更多”成为管理工作台底部导航的一部分。
5. 让权限、首页设置、账号管理都收拢到“更多”页，不分散到其他业务页面。
6. 严格控制边界，不改动客户侧、OCR、商品、订单主链。

## 3. 非目标

本 PRD 不做以下内容：

1. 不做微信快捷登录。
2. 不做客户侧账号体系改造。
3. 不做支付、物流、购物车、收藏、优惠券等新能力。
4. 不做找回密码闭环的第一版实现。
5. 不改现有商品、SKU、订单、库存、OCR 业务语义。
6. 不把管理域和客户域混成一套登录逻辑。

## 4. 交付约束

未来进入实现时，必须遵守以下束缚：

1. 前端、后端、数据、平台等作业时，只调用当前任务真正需要的技能或 MCP，不得为了“保险”额外堆工具，也不得调用与当前任务无关的能力。
2. 任何前端 UI 任务，除当前任务真正需要的技能或 MCP 外，必须额外调用 `ui-ux-pro-max` skill。
3. 需求拆解、设计、执行要分轮推进，每一轮都要先验证，再进入下一轮。
4. 每个模块完成后必须单独验证，不得把多个模块混成一次“已完成”。
5. 代码变更前必须先给出 Repository Impact Map 和 Execution Plan。
6. 不允许借本 PRD 扩展到其他业务线。

## 5. 信息架构

### 5.1 登录页

登录页是管理域的第一道门。

页面结构参考附件图：

- 左上返回箭头
- 大标题 `Login`
- 账号输入卡片
- 密码输入卡片
- `LOGIN` 主按钮
- `Forgot your password?` 可保留为视觉位，但第一版不接入找回流程

字段要求：

- 账号
- 密码

视觉要求：

- 白底
- 大留白
- 卡片式输入
- 黑色圆角主按钮
- 轻阴影

### 5.2 工作台底部导航

管理工作台底部导航固定为四项：

1. 工作台
2. 商品管理
3. 订单确认
4. 更多

其中“更多”是一级入口，不再作为附属菜单。

### 5.3 更多页

“更多”页只放三个管理模块：

1. 权限管理
2. 首页设置
3. 账号管理

## 6. 功能需求

### 6.1 账号密码登录

登录流程要求：

1. 当用户点击管理工作台入口且当前没有管理会话时，系统应显示登录页。
2. 当用户输入正确账号密码时，系统应创建管理会话并进入工作台。
3. 当账号或密码错误时，系统应停留在登录页并提示错误。
4. 当会话存在时，同一小程序运行周期内再次进入工作台不应重复登录。
5. 当小程序重新进入后，会话应失效并重新要求登录。

初始账号要求：

- 账号：`admin`
- 密码：`123456`
- 身份：创作者

### 6.2 工作台进入门禁

工作台门禁要求：

1. 任何管理域页面都必须先通过登录门禁。
2. 任何未登录深链进入管理页的情况，都应重定向到登录页。
3. 任何无权限账号进入受限模块时，都应显示无权限状态。
4. 工作台门禁是强校验，不得只靠页面隐藏按钮。

### 6.3 权限管理

权限管理要解决两件事：

1. 谁可以进入管理工作台。
2. 谁可以看到并操作哪些管理范围。

权限模型要求：

- 创作者拥有最高权限。
- 创作者可以授权店铺老板。
- 店铺老板可以在自己权限范围内授权员工。
- 员工只能使用被授予的范围。

权限范围至少应支持：

- 工作台进入权限
- 商品管理
- 订单确认
- 更多
- 首页设置
- 账号管理
- 权限管理

权限管理页应支持：

1. 查看账号列表。
2. 查看账号角色和权限范围。
3. 新增或授权账号。
4. 收回或禁用账号权限。
5. 保存授权操作记录。

### 6.4 首页设置

首页设置应支持以下可编辑项：

1. 背景更改
2. 标题更改
3. 宣传内容更改

约束：

- 仅影响首页展示，不改商品、订单、OCR 等业务语义。
- 编辑后应可预览。
- 保存后应立即反映到首页展示层。

### 6.5 账号管理

账号管理第一版只做一件事：

- 更改密码

规则：

1. 修改密码时必须校验旧密码。
2. 修改后当前会话失效。
3. 重新进入时必须使用新密码登录。
4. 第一版不做找回密码闭环。

## 7. UI 要求

### 7.1 登录页

登录页必须采用附件图的视觉语言：

- 白底
- 大标题
- 两个大输入卡
- 一个强主按钮
- 清晰留白
- 轻阴影

账号输入框应显示右侧格式通过标识或等价校验反馈。

### 7.2 更多页

“更多”页保持与高端女装管理工作台一致的极简高留白风格：

- 白底或浅灰底
- 圆角卡片
- 低噪声信息层级
- 单屏可快速定位三个入口
- 不做复杂装饰

## 8. 数据与角色边界

建议的管理域实体概念：

1. 管理账号
2. 角色
3. 权限范围
4. 首页设置
5. 授权记录

角色建议：

| 角色 | 能力 |
| --- | --- |
| 创作者 | 全量管理、授权老板、改密码 |
| 老板 | 管理自己范围内的模块、授权员工 |
| 员工 | 仅使用被授权模块 |

## 9. 模块拆分

后续实现必须按模块推进，一轮只做一个模块：

### 模块 1：登录页与会话

目标：

- 落地账号密码登录 UI
- 建立管理会话
- 支持初始账号 `admin / 123456`

验收：

1. 没登录时先看到登录页。
2. 正确账号密码可以进入工作台。
3. 错误密码会提示失败。
4. 会话在同一小程序周期内保持。

### 模块 2：工作台门禁

目标：

- 所有管理页统一走门禁
- 未登录自动回到登录页

验收：

1. 直接打开管理页会被拦截。
2. 会话失效后会重新登录。
3. 无权限访问时有明确提示。

### 模块 3：更多页与底部导航

目标：

- 底部导航增加“更多”
- “更多”页承载三个设置入口

验收：

1. 底部导航出现“更多”。
2. “更多”页只显示权限管理、首页设置、账号管理。
3. 其他业务页行为不变。

### 模块 4：权限管理

目标：

- 实现账号授权与范围控制

验收：

1. 创作者可授权老板。
2. 老板可授权员工。
3. 未授权账号不能进入受限模块。
4. 授权操作有记录。

### 模块 5：首页设置

目标：

- 允许修改首页背景、标题、宣传内容

验收：

1. 三项内容都可编辑。
2. 保存后首页可见变化。
3. 不影响商品与订单链路。

### 模块 6：账号管理

目标：

- 只做改密码

验收：

1. 旧密码校验有效。
2. 修改后旧密码失效。
3. 当前会话失效并要求重新登录。

### 模块 7：验证与文档同步

目标：

- 每轮都做验证
- 保持 PRD、实现与交付状态一致

验收：

1. 每个模块都有独立验证结果。
2. 不把 build smoke 当成人工验收。
3. 文档与实现状态同步。

## 10. 验证要求

实现阶段每个模块至少满足：

1. 单元或契约测试
2. 必要的集成测试
3. `pnpm.cmd run verify`
4. 必要时 `pnpm.cmd run verify:full`
5. 真机或开发者工具人工验收记录

## 11. 明确边界

以下内容在本 PRD 中保持不变：

1. 客户侧浏览商品不受影响。
2. 现有 OCR、草稿、商品、订单、库存主链不改语义。
3. 不把微信登录塞进管理工作台。
4. 不把管理域权限和客户域权限混成一套。
5. 不在未确认的情况下扩展找回密码、第三方登录、支付等新能力。
