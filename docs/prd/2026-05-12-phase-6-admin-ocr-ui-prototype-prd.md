# Phase 6 管理员 OCR UI 高保真原型 PRD

## 0. 文档定位

本文档是 `2026-05-08-enterprise-launch-master-prd.md` 中 Phase 6 “真实 OCR/AI 异步识别”的 UI 预览子 PRD。

本轮目标不是实现 Phase 6 业务能力，也不是重构小程序页面，而是先通过一个独立 HTML 高保真原型，让用户可以在浏览器中直观看到客户商城首屏、订单详情、客服对话、我的页面、右上角管理员入口、管理员登录、管理员 OCR 工作流的视觉风格、信息层级、交互节奏和异常状态。

本 PRD 对后续原型制作有强约束：原型必须与业务代码隔离，不能触碰现有小程序业务闭环，不能修改任何已完成的 domain、features、services、CloudBase、订单、库存、授权、图片存储或 OCR provider 实现。

## 1. 背景

主 PRD 已明确 Phase 6 的核心目标：

```text
把 Mock OCR 替换为真实 OCR/AI，同时保持页面和主 workflow 稳定。
```

Phase 6 涉及的能力包括：

1. OCR Job 状态机：`queued -> running -> succeeded`、`queued -> running -> failed`、`failed -> retrying -> running`。
2. 真实 OCR Provider：真实截图生成草稿、低置信度字段进入待人工确认、provider 失败有清晰提示。
3. 草稿质量与人工校正：记录字段置信度、字段来源截图、冲突字段、人工校正。
4. OCR 成本与限流：记录批次数量、provider 调用次数或成本、限制单次上传数量、异常高频调用告警。

当前 PRD 对业务目标已经清楚，但 UI 感受、页面视觉、管理员操作路径、异常反馈方式和高保真预览方式还不够明确。仅用文字描述无法判断界面是否符合“简约清新、交互流畅顺滑、适合微信小程序管理员操作”的要求。

因此需要先制作独立原型，让用户先看见 UI，再决定是否进入后续小程序页面实现。

## 2. 问题陈述

从用户角度看，当前存在以下问题：

1. 只听文字描述无法感受 UI 的颜色、间距、层级、动效和真实操作节奏。
2. “老板”角色文案不符合期望，管理员操作页面应统一显示“管理员”。
3. Phase 6 是真实 OCR/AI 链路，状态复杂，如果没有可视化原型，后续实现容易出现页面流程混乱。
4. Phase 7 才是正式产品级 UI 与设计系统重构，本轮如果直接改小程序页面，容易越界影响业务代码。
5. 当前必须严格保护已验收的主闭环、角色权限、订单库存、图片上传、CloudBase 和现有 Facade / ViewModel 边界。
6. 产品真实首屏应优先展示客户商城，而不是直接进入管理员工作台。
7. 管理员工作台应通过右上角入口进入，并在视觉原型中预留账号登录与权限管理路径。
8. 客户底部导航不得放管理员工作台或 OCR 识别入口。

## 3. 目标

### 3.1 产品目标

通过独立 HTML 高保真原型，明确 Phase 6 管理员 OCR 工作流的目标体验：

1. 管理员能直观看到截图上传、排队、识别中、识别成功、识别失败、重试、草稿复核的完整路径。
2. 管理员能快速识别哪些字段需要人工确认。
3. 管理员能理解失败原因，并看到可恢复动作。
4. 管理员能看到本批次图片数量、识别进度、失败数量、待复核数量和限流风险。
5. 用户能通过浏览器预览 UI 感受，再决定是否批准进入小程序实装。
6. 客户首先看到商城展示页，并能进入订单详情、客服和我的页面。
7. 管理员通过右上角入口进入登录页，再进入工作台。
8. 权限管理只做入口和视觉预留，不实现真实权限逻辑。

### 3.2 工程目标

1. 原型必须是独立交付物，不依赖小程序运行时。
2. 原型不得调用真实后端、CloudBase、OCR provider、上传服务或业务 Facade。
3. 原型不得修改 `src/`、`backend/`、`cloudfunctions/`、业务测试、构建配置或任何已有功能模块。
4. 原型只用于视觉评审和交互路径评审，不作为业务验收结果。
5. 原型确认后，才能拆出后续小程序 UI 实现计划。

9. UI 不得让重试重复创建草稿。
10. UI 不得把原型中的假数据写入任何真实存储。
11. 原型中的登录和权限管理不得被视为真实认证或授权实现。

## 5. 视觉方向

### 5.1 风格关键词

本 UI 采用“简约清新的微信小程序商城 + 管理员工作台”方向。

设计应具备：

1. 清爽：背景干净，颜色克制，不使用浓重渐变和强装饰。
2. 稳定：状态层级明确，让管理员知道当前批次处于哪个阶段。
3. 高效：管理员能快速定位异常字段和失败任务。
4. 温和：错误提示清楚但不过度惊吓，适合微信小程序日常操作。
5. 流畅：页面切换、状态变化和列表展开应有轻微动效。
6. 商城优先：首屏先服务客户浏览和商品转化，管理入口保持清晰但不抢占客户购物路径。

设计不应出现：

1. 紫色、蓝紫渐变、霓虹光效。
2. 大面积深色科技风。
3. 营销页式 hero。
4. 过多悬浮卡片和重阴影。
5. 花哨插画抢占业务信息。
6. 使用“老板”作为角色文案。

### 5.2 文案原则

1. 所有管理员操作页面统一使用“管理员”。
2. 页面标题、按钮、状态、错误提示全部中文优先。
3. 不使用英文术语替代中文操作，除非是技术状态标识的辅助信息。
4. 不使用夸张营销文案。
5. 失败提示必须给出下一步动作。

示例：

```text
商城
订单详情
客服
我的
右上角管理入口
管理员登录
管理员工作台
截图识别
识别任务
待人工复核
低置信度
字段冲突
重试失败项
查看草稿
```

## 7. UI/UX Pro Max 质量约束

本 PRD 后续原型必须额外遵守 `ui-ux-pro-max` 的移动端 UI/UX 质量门槛。该门槛优先用于约束高保真 HTML 原型，不代表进入小程序业务实装。

### 7.1 可访问性

1. 正文与背景对比度目标不低于 WCAG AA 的 4.5:1。
2. 状态信息不得只依赖颜色表达，必须同时有中文标签或说明。
3. 图标按钮如出现在原型中，必须有可见文字或可读说明。
4. 页面标题层级必须清楚，不能用纯视觉大小替代结构层级。
5. 错误、成功、重试、处理中状态必须能被文字理解。

### 7.2 触控与移动端交互

1. 主要按钮、列表操作、图片删除、重试、编辑等触控目标不得小于 44px x 44px。
2. 相邻可点击目标之间至少保留 8px 间距，避免误触。
3. 关键操作不得依赖 hover，必须通过点击或触摸完成。
4. 异步按钮必须有 loading / disabled / success / error 反馈。
5. 破坏性动作必须二次确认或提供清晰撤销路径。
6. 重要操作区域必须避开手机底部手势条和顶部安全区。

### 7.3 布局与响应式

1. 原型必须移动端优先，核心预览宽度为 375px。
2. 必须同时检查 390px 和 414px 宽度。
3. 不允许出现横向滚动。
4. 文本优先换行，不允许挤出按钮、标签、卡片或弹窗。
5. 固定底部操作区必须为内容预留底部空间，不能遮挡列表末尾。
6. 主内容优先显示当前任务、风险摘要和下一步动作，次要说明可折叠或下沉。

### 7.4 性能与感知速度

1. 原型不得引入重型前端框架或远程依赖。
2. 图片必须有固定尺寸或宽高比例，避免布局跳动。
3. 加载超过 300ms 的状态必须显示骨架屏、进度行或按钮 loading。
4. 不使用会导致频繁重排的动画属性，如 `width`、`height`、`top`、`left`。
5. 动效只允许使用 `opacity` 和 `transform` 为主。
6. 每个视图最多保留 1-2 个明显动效，不能让界面显得嘈杂。

### 7.5 表单与反馈

1. 所有可编辑字段必须有可见标签，不能只用 placeholder。
2. 字段错误必须显示在对应字段附近。
3. 多个错误同时存在时，顶部必须有风险摘要。
4. 低置信度、字段冲突、待补全必须在字段级可见。
5. 编辑后必须有“已人工校正”反馈。
6. 超时、限流、图片模糊、格式不支持等错误必须包含恢复路径。

### 7.6 导航与状态保持

1. 管理员从工作台进入识别任务、草稿复核、失败任务后，返回时应能理解自己来自哪里。
2. 原型中的页面切换必须保持方向一致，不能让前进和返回动效混乱。
3. 从失败任务重试后，应保留该批次上下文。
4. 从字段来源截图返回草稿时，应回到刚才编辑的草稿位置。
5. 任何弹窗或抽屉必须有明确关闭按钮。

### 7.7 原型视觉评审清单

原型交付前必须逐项自查：

1. 是否所有角色文案均为“管理员”，没有“老板”。
2. 是否一眼能看出主任务和主按钮。
3. 是否一眼能看出识别进度。
4. 是否一眼能看出失败项和待复核项。
5. 是否能在 375px 宽度下无横向滚动。
6. 是否所有点击目标适合手指操作。
7. 是否所有错误都有下一步。
8. 是否状态色不靠颜色单独传达含义。
9. 是否动效克制且有因果关系。
10. 是否没有任何真实业务调用或真实数据写入。

目标：让客户查看订单状态和商家处理进度。

3. 授权说明。

1. 进入订单详情。
2. 进入客服。
3. 查看授权说明。

### 8.6 管理入口登录页

目标：为管理员工作台建立视觉上的账号登录和身份边界。
3. 返回商城。

### 8.7 管理员工作台

目标：让管理员一进入系统就知道今天该处理什么。

关键内容：

1. 今日识别任务摘要。
2. 待人工复核数量。
3. 识别失败数量。
4. 正在识别批次数量。
5. 快捷入口：上传截图、查看待复核、查看失败任务。

交互：

1. 点击“上传截图”进入截图识别任务页。
2. 点击“待复核”进入草稿复核页。
3. 点击“失败任务”进入失败重试视图。

### 8.8 截图识别任务页

目标：让管理员完成上传前检查和提交识别。

关键内容：

1. 页面标题：截图识别。
2. 当前批次名称或创建时间。
3. 上传图片缩略图网格。
4. 单次上传数量限制提示。
5. 图片状态：待提交、已提交、识别中、成功、失败。
6. 主操作按钮：开始识别。

交互：

1. 选择图片后显示缩略图。
2. 超过单次限制时禁用开始按钮并显示原因。
3. 点击开始识别后进入识别进度状态。
4. 图片可删除，但识别开始后不可删除已提交图片。

### 8.9 OCR 识别进度与失败重试页

目标：让管理员明确知道识别在做什么、是否失败、能否重试。

关键内容：

1. 批次状态步骤条：
   - 已上传
   - 排队中
   - 识别中
   - 生成草稿
   - 待人工复核
2. 进度摘要：
   - 总图片数
   - 已识别
   - 失败
   - 待复核
3. 识别中骨架行。
4. 失败原因列表。
5. 重试失败项按钮。

交互：

1. 识别中使用轻动效，不使用大面积 loading spinner。
2. 失败项可展开查看原因。
3. 点击“重试失败项”后进入 retrying 状态。
4. 重试不允许重复创建草稿。

### 8.10 草稿复核与人工校正页

目标：让管理员优先处理有风险的字段，再确认生成商品和 SKU。

关键内容：

1. 顶部风险摘要：
   - 待补全
   - 低置信度
   - 字段冲突
   - 可确认
2. 草稿按风险优先排序。
3. 每个草稿只突出 Phase 6 相关核心字段：
   - 商品货号
   - 商品名称
   - 销售价
   - 规格
4. 低置信度字段使用警告背景。
5. 冲突字段使用错误背景。
6. 已人工校正字段使用成功标记。
7. 支持查看字段来源截图。

交互：

1. 点击字段进入编辑状态。
2. 编辑后字段标记为已人工校正。
3. 点击“查看来源”打开截图局部预览。
4. 所有必填和高风险项处理完后，批量确认按钮才可用。

## 9. 状态设计

### 9.1 OCR Job 状态

原型必须展示以下状态：

```text
queued：排队中
running：识别中
succeeded：识别成功
failed：识别失败
retrying：重试中
```

每个状态都必须有：

1. 中文状态名。
2. 视觉标签。
3. 管理员可理解的说明。
4. 下一步动作或等待提示。

### 9.2 字段质量状态

字段级状态包括：

```text
confirmed：已确认
low_confidence：低置信度
conflict：字段冲突
missing：待补全
corrected：已人工校正
```

### 9.3 空状态

必须覆盖：

1. 暂无截图。
2. 暂无识别任务。
3. 暂无待复核草稿。
4. 暂无失败任务。

空状态必须给出下一步动作。

### 9.4 错误状态

必须覆盖：

1. 网络失败。
2. OCR provider 超时。
3. 服务限流。
4. 单次上传超过限制。
5. 截图格式不支持。
6. 截图过于模糊。
7. 识别成功但字段置信度过低。

错误状态不得只显示“失败”，必须说明管理员能做什么。

## 10. 高保真 HTML 原型要求

### 10.1 原型形式

后续原型应以独立 HTML 页面实现，供浏览器预览。

推荐形式：

```text
docs/prototypes/phase-6-admin-ocr-ui/
-> index.html
-> styles.css
-> app.js
-> assets/
```

如为了更快预览，也可以先创建单文件 HTML，但必须保持与小程序源码隔离。

### 10.2 设备尺寸

原型必须至少模拟微信小程序手机视口：

```text
宽度：375px
高度：812px
```

同时应检查：

```text
宽度：390px
宽度：414px
```

文本不得溢出按钮、标签、卡片和列表项。

### 10.3 交互模拟

原型可以使用静态假数据模拟以下交互：

1. 切换页面。
2. 从上传进入识别中。
3. 从识别中进入待复核。
4. 展开失败原因。
5. 重试失败项。
6. 编辑低置信度字段。
7. 打开字段来源截图预览。

交互只能修改原型内存中的假数据，不得连接真实服务。

### 10.4 动效要求

动效必须轻量：

1. 页面切换使用 `opacity` 和 `transform`。
2. 列表进入使用轻微 stagger。
3. 按钮点击使用轻微缩放或下压。
4. 识别中使用骨架 shimmer 或进度行。
5. 不使用复杂 3D、不使用重粒子、不使用大面积持续动画。

## 15. 风险与控制

### 15.1 风险：原型被误认为实装

控制：

1. 原型文件必须放在文档或原型目录。
2. 原型不得 import 小程序源码。
3. 原型不得调用真实接口。
4. 最终说明必须标注“仅用于视觉预览”。

### 15.2 风险：UI 设计提前改变业务含义

控制：

1. 原型使用假数据。
2. 字段名称沿用当前业务语义。
3. 所有业务判断只用视觉模拟，不作为真实规则。
4. 后续实装时仍以 ViewModel / Facade 合约为准。

### 15.3 风险：Phase 6 和 Phase 7 混淆

控制：

1. 本 PRD 只覆盖 Phase 6 OCR 工作流 UI 原型。
2. 不定义完整产品级设计系统。
3. 不重构店员和客户页面。
4. 不把原型样式直接推广到全项目。
