# 全项目延迟治理 PRD

## 0. 文档定位

本 PRD 定义 `VX close systhem` 小程序从“功能可用”进入“响应稳定、可量化、可上线验收”的延迟治理方案。

它补充而不是替代以下既有文档：

- `docs/prd/2026-05-08-enterprise-launch-master-prd.md`
- `docs/prd/2026-05-19-enterprise-launch-risk-grouped-prd.md`
- `docs/prd/2026-05-25-product-management-operations-prd.md`
- `docs/contracts/page-facing-ui-contracts.md`
- `docs/architecture/page-ui-boundary-inventory.md`

本 PRD 不是一次视觉重构，也不是一次业务能力扩展。它只治理页面进入、数据加载、远程调用、图片解析、缓存刷新和用户可感知等待时间。

## 1. 背景

当前用户反馈的直接问题是：切换到商品管理页面后，商品信息很久才加载出来。

只读排查发现，商品管理页当前存在典型的延迟放大链路：

```text
页面 onShow
-> 刷新商品管理 ViewModel
-> listProducts
-> 对每个商品 listSkus(productId)
-> 对每个商品解析图片字段
-> 页面渲染
```

当商品数量为 N 时，商品管理首屏至少会产生 `1 + N` 条业务读取链路，叠加云函数调用、云数据库读取和云存储临时 URL 解析后，页面进入速度会随着商品数和图片数线性变差。

进一步检查后，这不是单页问题。当前多个页面都具备类似风险：

1. 列表页在前端组合多个远程结果。
2. 页面筛选、选择或重新进入时重新拉全量数据。
3. 图片列表页解析了超出首屏展示需要的图片字段。
4. 缺少统一的请求去重、短缓存、stale-while-revalidate 和加载态策略。
5. 自动测试覆盖了业务正确性，但尚未系统性约束远程调用次数、首屏等待预算和 N+1 回归。

## 2. 问题陈述

从用户视角看，当前小程序的部分页面“点了以后等很久才有内容”，尤其是商品管理这类管理端高频页面。用户无法判断是网络慢、接口慢、图片慢还是页面卡住。

从工程视角看，当前延迟问题有三类根因：

1. 数据合同偏细：页面级 ViewModel 需要的数据没有被后端一次聚合返回，导致前端通过多个接口拼装。
2. 图片解析偏重：列表页解析了不一定马上展示的图片 URL，且没有明确的临时 URL 缓存预算。
3. 体验兜底不足：页面缺少统一的加载、缓存复用、刷新中提示、失败重试和延迟指标采集。

## 3. 目标

1. 让高频页面首屏进入速度可量化、可回归测试、可人工验收。
2. 消除商品管理、客户商品列表、客户商品详情、草稿复核、店员补图、订单确认等页面的 N+1 读取模式。
3. 将页面级数据读取收敛到稳定的聚合接口或 page-facing facade，不让页面直接组合仓储细节。
4. 建立全项目统一的图片 URL 解析策略：列表只解析首屏需要的主图，详情图延迟或按需解析，临时 URL 可缓存但不得作为 canonical 数据保存。
5. 建立统一的页面短缓存、请求去重、刷新策略和加载态规范。
6. 建立性能验证门禁：代码评审、自动测试和人工验收都必须能看见延迟治理是否退化。

## 4. 非目标

本 PRD 不做以下内容：

1. 不新增购物车、支付、物流、优惠券、退款、客服或营销能力。
2. 不改变客户浏览免登录、下单才触发授权和手机号绑定的业务语义。
3. 不改变订单库存预占、防超卖、确认、取消和库存释放语义。
4. 不改变 OCR 草稿确认、商品/SKU 生成和店员补图主链。
5. 不让页面层直接访问 CloudBase 集合、repository、mockDb、OCR provider 或 upload service 实现。
6. 不为性能优化牺牲权限校验、数据一致性或库存审计。
7. 不把构建 smoke、自动化检查或接口计时冒充真实设备人工验收。

## 5. 成功指标

### 5.1 用户可感知指标

每个高频页面必须明确以下指标：

1. 页面进入后 300ms 内展示稳定的加载态、缓存内容或空态骨架。
2. 有可用缓存时，500ms 内展示上一版可用内容，并标记正在刷新。
3. 正常网络下，页面 fresh 数据 p75 在 1500ms 内完成，p95 在 3000ms 内完成。
4. 远程接口失败时，页面必须在 3000ms 内进入可解释失败态，并提供重试入口或返回路径。
5. 用户执行写操作后，按钮必须立即进入忙碌态，完成后只刷新受影响的数据范围。

### 5.2 工程指标

1. 列表页首屏远程业务调用数量必须为 O(1)，不得随商品、SKU、草稿、订单数量线性增长。
2. 商品管理首屏读取不得再出现“每个商品一次 `listSkus`”。
3. 列表页不得解析非首屏展示必需的详情图 URL。
4. 同一页面同一数据 key 的并发刷新必须去重。
5. 页面筛选、tab 切换、关键词过滤等纯展示行为优先本地计算，不得默认重新请求远程。
6. 每个新增聚合接口必须有契约测试覆盖返回字段、权限边界、空态、错误态和调用次数约束。

## 6. 延迟治理总框架

推荐采用“先量化，再聚合，再缓存，再体验兜底”的路线。

### 6.1 方案对比

#### 方案 A：只给页面加 loading 和缓存

优点：

1. 改动小。
2. 用户感知会有所改善。

缺点：

1. 真实远程调用数不降。
2. 商品和图片数量增加后仍会慢。
3. 难以从根上避免 N+1 回归。

结论：只能作为辅助，不能作为主方案。

#### 方案 B：只优化商品管理页

优点：

1. 可以快速解决当前最明显的痛点。
2. 范围清晰。

缺点：

1. 其它页面仍可能出现相同问题。
2. 后续每个页面都会重复造一套缓存和聚合规则。

结论：适合作为第一试点，但不能作为全项目方案。

#### 方案 C：全项目 page snapshot + 聚合接口 + 缓存门禁

优点：

1. 一次建立全项目性能治理标准。
2. 保持现有 page-facing ViewModel/facade 边界。
3. 可以用测试锁住 O(1) 远程调用预算。
4. 商品管理可以作为第一模块快速落地，再复制到其它页面。

缺点：

1. 需要按模块推进，不能一轮改完所有页面。
2. 要补充性能测试和契约测试。

结论：采用方案 C，并以商品管理作为模块 B 的第一个落地点。

### 6.2 可复用功能扩展路径

本 PRD 的长期价值不是只修一次商品管理延迟，而是沉淀一条以后新增客户侧和管理侧功能都能复用的建设路径。

以后新增“购物袋”“收藏”“我的”等客户侧模块时，必须先按以下路径设计，再进入代码实现：

```text
产品 PRD
-> 页面分级和性能预算
-> page-facing ViewModel / Facade 合同
-> page snapshot 数据合同
-> CloudBase / 后端聚合 action
-> repository / storage / auth 边界
-> 页面 state、缓存、加载态、失败态
-> targeted tests
-> verify / verify:full
-> WeChat DevTools 或真机人工验收
-> delivery log 和 handoff
```

任何新功能都不得先从 `.vue` 页面里直接拼接口、直接查集合、直接写仓储或临时塞状态开始。

### 6.3 新功能五层合同

每个新模块必须明确五层合同。

#### 第 1 层：产品语义合同

说明这个模块解决什么用户问题，哪些行为是业务事实，哪些行为只是 UI 展示。

示例：

1. 购物袋保存客户想购买的商品和 SKU，不等于创建订单，不预占库存。
2. 收藏保存客户喜欢的商品，不等于购物袋，也不影响库存和订单。
3. 我的模块展示客户身份、订单入口、收藏入口和设置入口，不直接承载商家管理能力。

#### 第 2 层：数据归属合同

说明数据属于哪个 domain、由哪个 repository/CloudBase action 读写、是否需要用户身份。

要求：

1. 客户私有数据必须绑定 customer/openid 或当前 customer session。
2. 客户私有数据不得被商品公共列表接口泄露。
3. 客户侧写操作必须经过 CloudBase/后端权限校验。
4. 页面不得直接生成最终业务 id，除非该 id 是纯页面临时 id。

#### 第 3 层：page-facing 合同

每个页面必须先定义页面能读什么、能传什么、不能做什么。

要求：

1. `.vue` 页面只消费 ViewModel 字段。
2. `.vue` 页面只调用 facade command。
3. 业务 eligibility、状态转移、价格、库存、权限判断必须放在 feature/facade 或服务端。
4. 合同必须写入 `docs/contracts/page-facing-ui-contracts.md` 或对应的新合同文档。

#### 第 4 层：page snapshot 合同

每个高频页面必须定义首屏 snapshot。

要求：

1. snapshot 必须服务首屏渲染，而不是返回整库数据。
2. 列表 snapshot 必须 O(1) 远程读取，不随条目数产生 N+1 action。
3. 详情 snapshot 只返回当前对象所需数据。
4. 写操作成功后必须明确失效哪些 snapshot。
5. 纯筛选、tab、搜索、排序优先本地计算。

#### 第 5 层：验证合同

每个新模块必须有能防止性能和边界退化的测试。

要求：

1. ViewModel/facade tests 验证业务可见行为。
2. CloudBase client tests 验证 action 合同。
3. CloudFunction/repository tests 验证权限、空态、失败态和聚合结果。
4. page state tests 验证缓存命中、请求去重、写后刷新和失败态。
5. 性能预算 tests 验证数据量增加时远程 action 数量不增加。

### 6.4 新客户侧模块的预设计方向

以下内容不是本 PRD 的立即实现范围，而是后续开 PRD 时必须沿用的延迟治理路径。

#### 购物袋

建议页面：

1. 商品详情中的“加入购物袋”轻量入口。
2. 独立购物袋列表页。

建议 snapshot：

```text
getShoppingBagSnapshot
```

首屏字段：

1. bag items。
2. 每项商品名、主图、选中 SKU、数量。
3. 当前价格和库存状态预览。
4. 是否可结算。
5. 失效原因，例如已下架、库存不足、规格不存在。

关键约束：

1. 加入购物袋不创建订单。
2. 加入购物袋不预占库存。
3. 结算前必须重新走服务端库存和商品状态校验。
4. 购物袋列表不得对每个 item 单独拉商品详情和 SKU。

#### 收藏

建议页面：

1. 商品详情中的收藏/取消收藏入口。
2. 我的模块中的收藏列表入口。

建议 snapshot：

```text
getFavoriteProductsSnapshot
```

首屏字段：

1. 收藏商品 id。
2. 商品名、主图、最低价、上架状态。
3. 收藏时间。
4. 商品不可见或已下架时的展示状态。

关键约束：

1. 收藏不影响库存、订单、商品状态。
2. 收藏列表不得泄露其它客户收藏。
3. 收藏状态在商品列表/详情中可作为附加状态，但不能让公共商品列表变成客户私有重查询风暴。

#### 我的

建议页面：

1. 客户身份状态。
2. 我的订单入口。
3. 我的收藏入口。
4. 购物袋入口。
5. 授权/手机号状态和基础设置。

建议 snapshot：

```text
getCustomerMineSnapshot
```

首屏字段：

1. 当前客户身份摘要。
2. 手机号绑定状态。
3. 待处理订单数量。
4. 收藏数量。
5. 购物袋数量。
6. 最近订单摘要。

关键约束：

1. 我的模块可以触发登录/手机号授权，但不得改变“客户浏览商品不触发登录”的合同。
2. 我的模块不得暴露商家管理入口，除非经过角色权限判定。
3. 统计数字必须由聚合接口返回，不得让页面并发调用订单、收藏、购物袋多个列表接口来拼。

## 7. 页面分级

### 7.1 P0 页面

P0 页面是用户高频进入、直接影响核心业务闭环的页面：

1. 客户商品列表。
2. 客户商品详情。
3. 商品管理。
4. 草稿复核。
5. 店员补图任务。
6. 订单确认。

后续客户侧“购物袋”“收藏”“我的”模块一旦进入实现 PRD，应默认按 P0 页面处理，除非用户明确降级为低频能力。

### 7.2 P1 页面

P1 页面是管理工作台和配置类页面：

1. 管理工作台首页。
2. 更多页。
3. 首页设置。
4. 账号管理。
5. 权限管理。
6. 登录页。

### 7.3 P2 链路

P2 链路是低频但上线风险较高的运维/后台链路：

1. CloudBase 生产配置检查。
2. OCR job 进度刷新。
3. 备份恢复和回滚演练。
4. 发布后观察和远程 smoke。

## 8. 模块 A：延迟观测与基线

### 8.1 目标

先建立全项目统一的延迟观测方法，避免凭感觉优化。

### 8.2 允许影响范围

允许：

```text
src/services/performance/
src/features/*/performance test helpers
scripts/
docs/testing/
docs/plans/
```

禁止：

```text
业务状态语义
订单、库存、授权、OCR、商品发布规则
页面直接访问 repository 或 CloudBase 集合
```

### 8.3 任务

1. 定义 `PageLoadTrace`：页面名、开始时间、缓存命中、远程 action 名、远程 action 耗时、图片解析耗时、结束时间、失败原因。
2. 定义 `RemoteCallTrace`：action、params shape、耗时、成功/失败、错误码，不记录 secret 和用户隐私。
3. 在本地测试中提供可注入计时器，避免真实时间导致 flaky。
4. 为 P0 页面建立当前调用次数基线。
5. 为后续模块建立对比方式：优化前调用数量、优化后调用数量、优化前耗时、优化后耗时。

### 8.4 验收

1. 可以在测试中断言某个页面 ViewModel 的远程 action 数量。
2. 性能日志不输出 secret、openid、手机号、真实 token、图片签名 URL query。
3. 文档记录每个 P0 页面的当前瓶颈和目标预算。

## 9. 模块 B：商品管理聚合接口试点

### 9.1 目标

把商品管理从 `listProducts + N * listSkus + N * imageResolve` 改为单次聚合读取。

### 9.2 新接口建议

新增商品管理快照接口：

```text
listOwnerProductCards
```

返回内容：

1. 商品基础字段。
2. `statusLabel`。
3. `skuCount`。
4. `canPublish`。
5. `publishBlockReasons`。
6. 列表主图可展示 URL。
7. `readyProductCount`。
8. `serverTime` 或 `snapshotVersion`。

接口内部允许：

1. 一次读取 products。
2. 一次读取 skus。
3. 按 `productId` 在云函数或后端聚合 SKU。
4. 对列表主图做批量临时 URL 解析。

接口内部禁止：

1. 对每个商品调用一次远程 `listSkus`。
2. 解析详情图数组中的所有图片。
3. 返回页面不需要的库存流水、订单、草稿等数据。

### 9.3 前端状态策略

1. 商品管理页首次进入时请求 `listOwnerProductCards`。
2. 状态筛选改为本地计算。
3. 30-60 秒内重新进入页面优先展示缓存，并后台刷新。
4. 上架、下架、删除、简介保存、库存保存后只刷新商品管理 snapshot。
5. SKU 库存弹层仍可按需读取单个商品 SKU 详情，不并入列表首屏。

### 9.4 验收

1. 商品管理首屏远程业务 action 数量不随商品数量增长。
2. 商品 1 个、20 个、100 个时，前端调用 `listOwnerProductCards` 的次数一致。
3. 状态 tab 切换不触发新的远程读取。
4. 列表页只解析主图，不解析详情图。
5. 原有单品上架、批量上架、下架、删除、简介、SKU 编辑业务结果不变。

## 10. 模块 C：客户商品列表与详情聚合

### 10.1 客户商品列表

当前客户商品列表已使用商品摘要接口，但仍会对每个商品解析完整图片字段。

优化目标：

1. 列表接口返回列表所需主图 URL 和最低价。
2. 列表页只使用主图、名称、货号、最低价和跳转 id。
3. 详情图在客户详情页按需解析。
4. 列表刷新支持短缓存和后台刷新。

验收：

1. 客户商品列表首屏远程业务 action 数量为 O(1)。
2. 列表页不解析 `imageUrls` 全量详情图。
3. 商品上架/下架后，客户列表最终一致刷新，不展示已下架商品。

### 10.2 客户商品详情

当前客户详情会读取已发布商品列表，再找单个商品，再读取 SKU。

优化目标：

1. 新增或扩展单商品详情接口：

```text
getPublishedProductDetail(productId)
```

2. 单次返回已发布商品、SKU 列表、详情图可展示 URL、默认选中状态所需数据。
3. SKU 选择只在本地切换，不重新请求详情。
4. 下单前仍重新走服务端库存校验和订单创建，不能相信页面缓存库存。

验收：

1. 进入商品详情不再拉全量已发布商品。
2. SKU 点击不触发远程读取。
3. 下单仍以服务端最新库存为准。
4. 售罄 SKU 可查看但不可下单的语义保持不变。

## 11. 模块 D：草稿复核、店员补图与订单确认快照

### 11.1 草稿复核

优化目标：

1. `getLatestDrafts` 继续作为核心读取，但返回结构应适合页面直接渲染。
2. 低置信度、字段来源、价格冲突、可确认状态可以在 feature/facade 聚合，页面不得重复计算。
3. 编辑单条草稿后优先局部更新或刷新当前批次 snapshot，避免全项目数据刷新。

验收：

1. 草稿复核页面首屏远程 action 数量为 O(1)。
2. 编辑字段不导致重复拉取无关商品、SKU、订单数据。
3. 低置信度和冲突提示不因缓存而丢失。

### 11.2 店员补图任务

当前店员补图同时读取 OCR 批次和待补图商品。

优化目标：

1. 聚合为 `getStaffImageTaskSnapshot`，一次返回批次选项和待补图商品。
2. 关键词和批次筛选优先本地计算。
3. 补图上传完成后只刷新待补图商品 snapshot。

验收：

1. 关键词输入不触发远程读取。
2. 批次切换不触发远程读取，除非缓存过期或用户手动刷新。
3. 店员仍不能上架商品或编辑商品运营信息。

### 11.3 订单确认

当前订单确认读取 `listMerchantOrders`，短期风险低于商品页，但仍需要统一缓存和加载态。

优化目标：

1. 订单列表首屏保持 O(1) 读取。
2. 确认/取消订单后只刷新订单 snapshot。
3. 不缓存写操作结果为最终真相，仍以服务端返回订单为准。

验收：

1. 订单确认/取消按钮立即进入忙碌态。
2. 操作完成后订单状态、库存释放语义和错误提示保持不变。
3. 订单页面不直接做状态流转判断以外的业务写入。

## 12. 模块 E：图片 URL 与资源加载治理

### 12.1 目标

减少图片临时 URL 解析造成的远程等待，同时保护 `cloud://` 文件 ID 作为 canonical 数据的原则。

### 12.2 策略

1. 数据库存储继续保留 `cloud://` 文件 ID，不保存已签名临时 URL。
2. 列表页只解析主图。
3. 详情图只在详情页、图片预览或编辑弹层按需解析。
4. 同一次页面快照中对图片 ID 去重后批量解析。
5. 临时 URL 缓存 TTL 必须短于 CloudBase `maxAge`，建议 45-55 分钟。
6. 缓存 key 使用 `cloud://` file ID，不使用签名 URL。
7. 签名 URL 过期或解析失败时显示占位图和重试入口，不阻塞非图片信息展示。

### 12.3 验收

1. 商品列表、商品管理、店员补图列表不会解析详情图数组。
2. 同一图片 ID 在一次刷新中只解析一次。
3. 临时 URL 不写回商品主档。
4. 图片失败不导致商品文字信息消失。

## 13. 模块 F：统一页面缓存与刷新策略

### 13.1 目标

让页面切换更快，同时不牺牲业务正确性。

### 13.2 缓存模型

建议引入轻量 `PageSnapshotCache`：

```text
key
data
createdAt
expiresAt
staleAt
inFlightRequest
lastError
```

缓存类型：

1. `fresh`：可直接展示。
2. `stale`：可先展示，但必须后台刷新。
3. `expired`：不可作为最终展示，只能作为骨架或兜底。
4. `inFlight`：已有同 key 请求，后续调用复用 Promise。

### 13.3 页面规则

1. onShow 不等于必然全量远程刷新。
2. 用户从详情返回列表时，优先展示缓存。
3. 写操作成功后显式失效相关 snapshot。
4. 写操作失败不得污染缓存。
5. 纯本地筛选、搜索、tab 切换不得默认失效缓存。
6. 页面离开后不再把旧请求结果覆盖到新页面状态。

### 13.4 验收

1. 同一页面连续 onShow 不产生重复并发请求。
2. 缓存过期后会后台刷新。
3. 写操作后能看到最新结果。
4. 失败态不会把旧缓存误报为新数据。

## 14. 模块 G：加载态、失败态与用户感知优化

### 14.1 目标

即使网络不稳定，也让用户明确知道页面正在加载、哪些内容可用、哪里失败、如何重试。

### 14.2 UI 行为

1. P0 页面必须有首屏加载态或骨架屏。
2. 有缓存时显示缓存内容，并展示轻量刷新状态。
3. 空态、失败态和权限态必须区分。
4. 写操作按钮必须有忙碌态和防重复提交。
5. 失败消息必须用户可理解，不能泄露数据库、CloudBase 内部错误或 secret。
6. 重试按钮必须只重试当前 snapshot，不触发无关业务链路。

### 14.3 UI Skill 准则

如果实现阶段修改任何页面视觉、交互、布局、按钮状态、骨架屏、错误态或加载态，必须按项目 AGENTS 规则先读取并应用：

1. `design-taste-frontend`
2. 如涉及高端视觉质量，追加 `high-end-visual-design`
3. 如重做既有页面，追加 `redesign-existing-projects`

纯后端聚合、缓存服务、测试或文档任务不需要调用 UI skill。

## 19. 开放问题

以下问题不阻塞模块 0 和模块 1：

1. 最终性能预算是否需要按真机型号分档。
2. CloudBase 临时 URL 缓存 TTL 是否采用 45 分钟还是 55 分钟。
3. 是否需要增加生产只读性能 smoke，定时检查 P0 action p95。
4. 是否需要为管理端增加手动“刷新当前页”入口。
5. 购物袋、收藏、我的模块的最终导航结构是否独立成 tab，还是先从商品详情和我的页入口进入。

暂定决策：

1. 先按正常网络 p75/p95 建立预算。
2. 图片 URL 缓存 TTL 暂按 45 分钟设计。
3. 模块 1 只解决商品管理，不扩展订单、支付、物流等能力。
4. 每个模块结束后更新 delivery log，再进入下一个模块。
5. 购物袋、收藏、我的模块不混入本 PRD 的模块 0-6 实施，后续分别开 PRD，但必须复用本 PRD 的五层合同和 snapshot 路径。
